## 為什麼可能省 token 成本

這是 Polaris 實驗最核心的直覺。

傳統做法通常是：

每一次都把很長的背景、規則、上下文、任務說明，全部塞進模型。

這很貴。

因為模型每次都要重新讀一大包文字。

Polaris 的實驗方向比較像這樣：

1. 先把任務拆開
2. 把任務變成結構
3. 把結構壓成一張「任務地圖」
4. 讓模型根據這張地圖完成輸出

這裡的「地圖」可以理解成拓樸。

拓樸不是記住每一個字，而是保留任務之間的關係。

| 傳統長上下文做法 | Polaris 拓樸做法 |
| --- | --- |
| 每次把整份說明塞給模型 | 先建立任務結構 |
| 模型每次重新讀長文 | 模型讀更短的結構化表示 |
| 成本跟上下文長度一起變大 | 成本可以下降 |
| 容易因為長文雜訊漂移 | 結構比較容易被固定 |
| 很像搬一整座圖書館 | 比較像給一張精準地圖 |

白話例子：

假設你要叫一個人去城市裡找東西。

傳統方式像是每次都把整本城市百科全書交給他。  
Polaris 的方向像是先畫出路線圖、區域關係、重要地標，再叫他照圖行動。

你不需要每次搬整本百科全書。

你需要的是保留正確結構。

<details>
<summary>展開：這不是單純把提示詞變短</summary>

如果只是把提示詞剪短，通常會丟失資訊，品質也會下降。

Polaris 的實驗重點不是粗暴刪字，而是把任務內容重新整理成更穩定的結構。

可以想像成：

| 原本長文字 | 轉換後 |
| --- | --- |
| 一大段任務說明 | 任務節點 |
| 很多規則 | 規則關係 |
| 多個目標 | 目標順序 |
| 多種限制 | 邊界條件 |
| 可能錯誤 | 審核點 |
| 最後輸出 | 結果格式 |

這樣模型收到的不是一團很長的文字，而是一個比較清楚的任務結構。

模型接下來要做的事，比較像是：

讀懂這張任務地圖。  
把地圖翻譯回正確輸出。  
完成題目要求。

所以你可以把它理解成：

> 先把任務變成結構，再讓模型把結構翻譯成答案。

這也是為什麼我們會觀察 token 成本下降，但結果品質仍然能保留的原因之一。

</details>

main run 記錄了 720 of 720 outputs、100 percent 解析成功、0 硬性紅線、0 警告、0 程式庫幻覺、0 把干擾來源當正式來源，並且達到 `FULL_SANDBOX_STRONG_PASS`。

白話解讀：

PP02C 不宣稱所有程式任務都已經解決。它顯示的是，這個特定程式修復與合約驗證分支，通過了目前記錄下來的沙盒、硬性紅線與干擾來源檢查。

</details>

## 黑粉 FAQ

這一段是給第一次看到 Polaris 實驗證據包的讀者。

我們知道這些結果看起來很猛烈，所以也很正常會有人懷疑。下面先把常見質疑整理出來，方便大家快速理解這批公開資料到底代表什麼，以及還不代表什麼。

<details>
<summary>Q1. 這些數據會不會是假的？</summary>

這是最合理的第一個問題。

所以我們不是只放截圖，也不是只放一張漂亮分數表。這次公開的是完整證據包，裡面包含模型原始輸出、整理後結果、每題判定、token 成本紀錄、審核紀錄與 SHA256 檔案指紋。

簡單說，如果只是想做假，通常只會放最後分數。但這裡放的是一整條實驗軌跡，讓讀者可以往回看模型實際輸出了什麼，結果如何被整理，最後如何被判定。

</details>

<details>
<summary>Q2. 會不會只是挑好看的結果公開？</summary>

這也是合理懷疑。

目前公開的 PP02A 就保留了一個前一版較弱的參考紀錄，不是只放最後漂亮版本。PP02C 也保留沙盒檢查、硬性紅線檢查與干擾來源檢查。

我們希望公開的是「實驗過程」，不是只公開勝利截圖。

當然，這仍然是第一版公開證據層，不是最終完整 benchmark。更完整的數學邏輯、重現流程與後續資料，會在 2026/05/05 開源發布後逐步補上。

</details>

<details>
<summary>Q3. 這是不是只是把提示詞寫短，所以 token 變少？</summary>

不是。

如果只是把提示詞剪短，通常會丟失資訊，品質也會下降。

Polaris 的核心方向不是粗暴縮短文字，而是先把任務拆解、結構化，再轉成更像「任務地圖」的形式。模型不是只收到一段變短的文字，而是收到比較清楚的結構關係。

白話說：

傳統做法像是每次都搬一整本百科全書給模型看。  
Polaris 的方向比較像是先畫好地圖，再讓模型照著地圖完成任務。

這就是為什麼 PP01 會特別看 token 比例與輸出品質是否同時保留。

</details>

<details>
<summary>Q4. 你們是不是偷把答案放進提示詞裡？</summary>

目前公開資料中有保留洩漏檢查與審核紀錄。

例如 PP01 記錄了 leakage count 為 0，也有錯誤來源與過度宣稱相關檢查。PP02C 也保留了來源行為、干擾來源、metadata as evidence 等檢查。

這些檢查的目的，就是避免模型靠偷吃答案、亂引用來源、或把不該當證據的東西當證據。

這不代表未來所有實驗永遠不會出錯，但目前公開包至少把這些檢查結果放出來，而不是只放最後分數。

</details>

<details>
<summary>Q5. 為什麼現在不直接放 Colab 和完整數學？</summary>

因為這次是第一層公開證據，不是完整執行包。

目前先公開的是證據資料：模型原始輸出、整理後結果、判定表、token 成本、審核紀錄與檔案指紋。

完整數學邏輯、執行 notebook、Colab 重現流程，預計在 2026/05/05 開源發布後陸續補上。

這樣做的原因很簡單：

先讓大家看到實驗證據，再公開更深層的執行與數學材料。

</details>

<details>
<summary>Q6. 這些結果是不是代表小模型一定可以打敗大模型？</summary>

不是。

目前比較精準的說法是：

在已測試的 Polaris 任務裡，結構化後的任務表示可以在某些分支中大幅降低 token 成本，同時保留強輸出表現。

這不等於所有小模型都能打敗大模型，也不等於所有任務都已經解決。

PP01 支持的是「在已測試任務裡，結構化任務表示能保留強輸出表現並降低 token 成本」。這是一個有範圍的結論，不是宇宙無限宣稱。

</details>

<details>
<summary>Q7. 這些實驗是不是已經證明所有數學、程式、代理任務都解決了？</summary>

沒有。

目前公開資料只支持目前已測試分支的結果。

PP02B 是偏數學任務的公開證據包，但它不宣稱所有數學都解決。  
PP02C 是程式修復與合約驗證的公開證據包，但它不宣稱所有程式任務都解決。

我們比較嚴謹的說法是：

這些包展示了目前分支在指定測試條件下的結果，並且保留了可檢查的證據鏈。

</details>

<details>
<summary>Q8. SHA256 是做什麼用的？</summary>

SHA256 可以理解成檔案指紋。

如果一個 zip 檔被改過，SHA256 通常就會改變。

所以我們放 SHA256，是為了讓讀者可以確認自己下載的檔案，是否和公開紀錄一致。

白話說：

它不是拿來證明實驗一定正確，而是拿來確認檔案有沒有被偷偷改過。

</details>

<details>
<summary>Q9. 如果我不是工程師，我應該看哪裡？</summary>

建議先看每個壓縮包裡面的 README，然後看最終判定檔，再看 token 成本與審核紀錄。

最簡單的讀法是：

先看這個實驗測什麼。  
再看它有沒有通過。  
再看它省了多少 token。  
最後看有沒有洩漏、警告或硬性錯誤。

如果只想快速理解，可以先看 PP01，因為它是目前最大、最乾淨的大規模公開證據包。

</details>

<details>
<summary>Q10. 這次公開最重要的意義是什麼？</summary>

最重要的不是「我們放了一堆 zip」。

最重要的是：

我們先公開證據，再公開更深層的數學與執行材料。

這代表讀者不必只看截圖或相信口號，而是可以先看到模型原始輸出、整理後結果、判定表、token 成本、審核紀錄與檔案指紋。

這是 WFGY 5.0 Polaris Protocol 在 2026/05/05 開源發布前的第一層公開證據。
