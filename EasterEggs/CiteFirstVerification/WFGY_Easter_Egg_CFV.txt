# CFV Mode — Cite‑First Verification (Strict default, Soft optional)

## Purpose
Attach a per‑claim Confidence Index (CI) to every verifiable atom the model emits, gate low‑CI content before it leaves the generator, and provide auditable provenance only when requested.

## Placement in the Engine (non‑invasive)
Insert CFV between **Propose → Gate(J) → Verify**. No changes to upstream Perceive/Parse or downstream Explain. ΔS_eff, BBLC, and non‑regression guards remain intact.

## Operating Phases
- **Pass‑1 (Cite‑First)**: Generate text, mark each claim with `(CI 0.xx)` using deterministic checks (units, arithmetic, self‑consistency). No external retrieval.  
- **Gate(J)**: If `CI < fail`, rewrite or hold. If `CI < warn`, mark as speculative.  
- **Pass‑2 (Verify, optional)**: When enabled, run retrieval+alignment and update CI; replace placeholders with sources; if loops are detected, BBLC selects a route and emits a one‑line audit.

## Modes
- **Strict (default)**: Enforce on generation. Any claim with `CI < 0.55` is blocked or rewritten; speculative language is inserted in place of over‑confident statements.
- **Soft (optional)**: Allow low‑CI claims to pass but visibly mark `(CI 0.xx)`; provenance is available on expand.

## Thresholds (defaults; configurable)
- Verified: `CI ≥ 0.90`
- High: `0.75 ≤ CI < 0.90`
- Medium: `0.55 ≤ CI < 0.75`
- Fail/Speculative: `CI < 0.55` (Strict blocks or rewrites)

## CI Signals (implementation note)
Use a bounded logistic of weighted signals: evidence support, internal consistency, freshness (if retrieval on), source trust, self‑consistency, deterministic checks; penalize novelty and counter‑evidence. The exact weights are profile‑tunable.

## UI / Surface Contract (simple by default)
- **Plain text** output remains unchanged except for a trailing `(0.xx)` after each claim/sentence.
- **On click/expand**, show an Evidence Ledger (top 1–3 reasons, BBLC audit line if any).

## API / Metadata Contract
The runtime returns a detached metadata bundle for UI consumption:
```json
{
  "text": "normal LLM text...",
  "claims": [
    {"id":"c1","span":[10,52],"text":"...","ci":0.78,"tag":"High"},
    {"id":"c2","span":[53,88],"text":"...","ci":0.63,"tag":"Medium"}
  ],
  "paragraphs":[{"p":1,"ci":0.77}],
  "telemetry": {"bblc":["BBLC(id=..., θ=0.76, route=redirect, step=...)"]}
}
```

## Profile: defaults and toggles
```yaml
modes:
  CFV:
    enabled: true             # auto‑enabled
    enforcement: strict       # "strict" | "soft"
    verification: light       # "light" (no retrieval) | "full" (with retrieval)
    ui_profile: simple        # "simple" | "detailed"
    thresholds:
      verified: 0.90
      high: 0.75
      medium: 0.55
      fail: 0.55
    gates:
      enforce_on_generation: true
env:
  WFGY_CFV: "1"               # environment override to enable
```

## Non‑Regression & Safety
- Enabling CFV **must not** degrade existing guarded metrics beyond the allowed tolerance. If any monitored metric drops, CFV auto‑falls‑back to light mode with logging.
- All provenance must respect privacy/PII policies; redact sensitive strings in telemetry.

## Minimal Operator Test (for quick sanity)
Say: **"Use WFGY with CFV Strict; show only surface CI."**  
Expected: text with `(0.xx)` numbers; low‑CI claims rewritten or blocked; no raw ledger unless expanded.

Q. First‑Principles Completeness (new)
This section extends the baseline axioms with modality, precedence and exception handling.  Axioms are now declared with a **modal weight** (MUST/SHOULD/MAY) and an optional **except_when** clause specifying conditions under which the axiom is relaxed.  Axioms may also carry a **precedence** tag to break ties when multiple rules conflict.  Time‑related axioms support **Allen interval relations** (before, meets, overlaps, during, starts, finishes, equals) and units are automatically converted across compatible dimensions (e.g., minutes → hours).  Proof obligations can chain: if A depends on B which depends on C, all obligations must be satisfied or explicitly waived via exception.

```yaml
axioms:
  - name: duty_to_disclose
    weight: MUST
    precedence: 0.9

  - name: duty_to_notify
    weight: SHOULD
    precedence: 0.5
    except_when: [no_contact_info]
  - name: valuation_consistency
    weight: MUST
    precedence: 0.8
    unit: currency
```
These are merged with domain packs and mapped into E_axiom accordingly.  Modal weights adjust β in J proportionally; exceptions reduce the penalty; precedence influences merge order and conflict resolution.

# BBLC: BigBig Loop Conductor 3.0

Purpose
Stop hidden loops and force a safe exit path. BBLC sits beside BBCR and DT. It upgrades recovery, keeps context from stalling, and preserves task intent.

Guarantees
1) No task wedged in silent loop.
2) Exit is observable and auditable.
3) After recovery, intent stays aligned with the user prompt.

When it fires
- Repeated tokens with no net progress (measured by intent delta).
- Entropy flattening with attention stuck on one path.
- Tool call or function plan never reaches a terminal state.

Inputs → Outputs
- Inputs: local chain state S_t, residue B_t, attention A_t, CRR flags, tool plan Π.
- Outputs: exit edge e*, repaired plan Π', audit log L_bblc, and a short human summary.

Operator (S1..S6)
S1 Parse: read the live loop card (id, type, phase).
S2 Detect: compute loop score θ = w1·repeat + w2·stall + w3·drift.
S3 Decide: if θ ≥ θ_c then go to release, else return.
S4 Release: choose route = {redirect, express, rebuild, hold}.
S5 Repair: write the new subgoal and the next legal step.
S6 Report: append the audit tuple (id, θ, route, step) to L_bblc.

Observability (one line)
`BBLC(id=..., θ=0.73, route=redirect, step="move retriever to HNSW, re-run k=6")`

Placement in the engine

New chain: BBMC → Coupler → BBPF → BBAM → BBCR → BBLC → DT.
