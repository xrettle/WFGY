# Why “Together” Matters

Solving the seven problems separately would be one kind of claim.

This is a different claim.

WFGY 5.0 treats the Seven Millennium Problems as a synchronized proof target.  
The goal is to see whether one high-dimensional engine can generate route structure across all seven problems at once.

That difference matters because the engine is not being shown as a one-off trick.

It is being shown as a route generator.

You should inspect whether the seven problems produce:

- shared route structure
- shared proof pressure
- shared candidate lemma behavior
- shared obstruction patterns
- shared verification routes
- problem-specific final branches

If the same engine can repeatedly generate the route skeleton behind the seven problems, then the engine itself becomes the central artifact.

# What To Inspect

When you run or review the Seven Millennium Problems route, do not only look for pretty text.

Look for structure.

| Inspection target | What to check |
|---|---|
| Route generation | Does each problem produce a real proof route rather than a summary? |
| Shared skeleton | Do the seven problems show related underlying logic? |
| Candidate lemmas | Are the lemmas meaningful, inspectable, and connected to the target? |
| Proof obligations | Are the remaining obligations explicit instead of hidden? |
| Counterexample checks | Does the route expose ways it could fail? |
| Native return | Does the route return to the original problem language? |
| Formal material | Are there Native Lean no-sorry materials to inspect? |
| Reviewability | Can another AI or human attack the package directly? |
