## 這套方法為什麼可能省 token 成本

這是最核心的直覺。

傳統做法通常是：

每一次都把很長的背景、規則、上下文、任務說明，全部塞進模型。

這很貴。

因為模型每次都要重新讀一大包文字。

Polaris 的實驗方向比較像這樣：

先把任務拆開。  
再把任務變成結構。  
再把結構壓成一張「任務地圖」。  
接著讓模型根據這張地圖去完成輸出。

這裡的「地圖」可以理解成拓樸。

拓樸不是在記住每一個字，而是在保留任務之間的關係。

例如：

| 傳統長上下文做法 | Polaris 拓樸做法 |
| --- | --- |
| 每次把整份說明塞給模型 | 先建立任務結構 |
| 模型每次重新讀長文 | 模型讀更短的結構化表示 |
| 成本跟上下文長度一起變大 | 成本可以下降 |
| 容易因為長文雜訊漂移 | 結構比較容易被固定 |
| 很像搬一整座圖書館 | 比較像給一張精準地圖 |

白話例子：

假設你要叫一個人去城市裡找東西。

傳統方式像是每次都把整本城市百科全書交給他。  
Polaris 的方向像是先畫出路線圖、區域關係、重要地標，再叫他照圖行動。

你不需要每次搬整本百科全書。

你需要的是保留正確結構。

## 這不是單純把提示詞變短

這點很重要。

如果只是把提示詞剪短，通常會丟失資訊，品質會掉。

Polaris 的實驗重點不是單純變短，而是：

把任務內容重新整理成更穩定的結構。

也就是說，它不是粗暴刪字，而是把任務轉成一種更像「拓樸圖」的形式。

可以想像成：

| 原本長文字 | 轉換後 |
| --- | --- |
| 一大段任務說明 | 任務節點 |
| 很多規則 | 規則關係 |
| 多個目標 | 目標順序 |
| 多種限制 | 邊界條件 |
| 可能錯誤 | 審核點 |
| 最後輸出 | 結果格式 |

這樣模型收到的不是一團很長的文字，而是一個比較清楚的任務結構。

模型接下來要做的事，比較像是：

讀懂這張任務地圖。  
把地圖翻譯回正確輸出。  
完成題目要求。

所以你可以把它理解成：

> 先把任務變成結構，再讓模型把結構翻譯成答案。

這也是為什麼我們會觀察 token 成本下降，但結果品質仍然能保留的原因之一。
